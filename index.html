Executable Contracts: Eliminating Integration Drift in Multi-Agent AI Factories

How Specmatic contract testing acts as a strict compile-time check for AI agents, preventing silent API hallucinations and type mutations.

As AI coding assistants increasingly replace traditional development roles, a major architectural challenge has surfaced: **API Integration Drift**. When multiple AI agents work independently on different microservices, they lack the organic communication channel of human teams. Consequently, they make silent, divergent assumptions about payload structures, data types, and API names.

In this post, we present our research on using Executable API Contracts via Specmatic. By running a 5-phase agent simulation orchestrating 5 specialized agents over 4 microservices, we demonstrate how contract testing captures 100% of integration failures during development, saving over a hundred hours of debugging time.

Multi-Agent Simulation Architecture

Our environment simulates an autonomous engineering team. We deploy five distinct AI agents utilizing NVIDIA NIM's reasoning model meta/llama-4-maverick-17b-128e-instruct:

  • API Design Agent: Generates and updates OpenAPI contract files.
  • Backend Agent: Writes Express controllers and determines API responses.
  • Frontend Agent: Simulates consumer requests and mock data bindings.
  • Testing Agent: Builds test cases matching expected JSON schemas.
  • Documentation Agent: Generates markdown documents detailing the endpoints.

These agents develop and maintain four microservices: the User Service, Task Service, Notification Service, and Analytics Service.

The Resiliency Testing Gap

A vital learning from our experiment is that structural validation of static payloads is not enough. In our initial design, we created a lightweight structural validator that parsed JSON payloads and checked field names and types.

However, when Specmatic ran active resiliency tests against the running services—deliberately mutating values to null, changing strings to integers, and injecting unexpected properties—the application services failed 45 of 76 tests on the /tasks endpoint alone.

Why did this happen? Express controllers commonly read incoming JSON without validating type limits or null-safety. If the frontend sends a type it doesn't expect, the database driver crashes or silently corrupts the record. Only active runtime contract testing reveals these gaps.

Case Study: Resolving Task Service Resiliency

To make the Task Service resilient to type mutations and unexpected payloads, we updated the Express controller code. Look at the tabbed comparison below to see how validation logic was hardened to meet the contract. 

// services/task-service/src/controllers.ts (Pre-Fix)
export function createTask(req: Request, res: Response): void {
  const body = req.body;

  // Basic check - doesn't guard against arrays, null, or primitives
  if (!body || typeof body !== 'object') {
    res.status(400).json({ error: 'Invalid body' });
    return;
  }

  const { title, description, assigneeId } = body;

  // Omission check only - does not reject 'null' values!
  if (title === undefined || description === undefined || assigneeId === undefined) {
    res.status(400).json({ error: 'Missing required fields' });
    return;
  }

  // Database insertion goes ahead with unchecked types
  const task = createTaskRecord({ title, description, assigneeId });
  res.status(201).json(task);
}
// services/task-service/src/controllers.ts (Hardened Post-Fix)
export function createTask(req: Request, res: Response): void {
  const body = req.body;

  // Reject null, array, or primitives explicitly
  if (body === null || body === undefined || typeof body !== 'object' || Array.isArray(body)) {
    res.status(400).json({ error: 'Request body must be a JSON object', code: 400 });
    return;
  }

  // Reject unexpected parameters (prevents schema drift)
  const unexpectedFields = Object.keys(body).filter(k => !ALLOWED_CREATE_FIELDS.includes(k));
  if (unexpectedFields.length > 0) {
    res.status(400).json({ error: `Unexpected fields: ${unexpectedFields.join(', ')}`, code: 400 });
    return;
  }

  const { title, description, assigneeId, priority } = body;

  // Validate required fields are present AND not null
  const missing: string[] = [];
  if (title === undefined || title === null) missing.push('title');
  if (description === undefined || description === null) missing.push('description');
  if (assigneeId === undefined || assigneeId === null) missing.push('assigneeId');
  if (missing.length > 0) {
    res.status(400).json({ error: `Missing required fields: ${missing.join(', ')}`, code: 400 });
    return;
  }

  // Strict type checking matching OpenAPI schemas
  if (typeof title !== 'string') {
    res.status(400).json({ error: 'Field "title" must be a string', code: 400 });
    return;
  }
  if (typeof description !== 'string') {
    res.status(400).json({ error: 'Field "description" must be a string', code: 400 });
    return;
  }
  if (typeof assigneeId !== 'number' || !Number.isInteger(assigneeId)) {
    res.status(400).json({ error: 'Field "assigneeId" must be an integer', code: 400 });
    return;
  }

  const task = createTaskRecord({ title, description, assigneeId, priority });
  res.status(201).json(task);
}

By updating validation to actively check parameters, data types, and null conditions matching the contract definitions, the task service went from 45 failed tests to 0 failures.

Key Engineering Learnings

  1. Contracts form a strict interface for Agents: In independent multi-agent environments, OpenAPI specifications act as a strict compile-time interface. If the prompt context contains contract specifications, model hallucinations drop dramatically.
  2. Resiliency Testing is the core value: Catching field-name drifts catches basic errors, but checking schema resilience (null-handling, type safety, enum checking) reveals deep-rooted security and reliability errors in microservices.
  3. Automatic combinatorics beats manual QA: Specmatic automatically generated 76 test cases for 2 endpoints by mutating inputs. Creating these tests manually would take substantial engineering hours, and they are easily missed by human testers.
  4. Continuous Feedback Loops: Programmatic execution of Specmatic CLI tests inside the build orchestration cycle allows agents to receive live, structured error messages when they generate drift. They can self-heal and resolve issues autonomously.

External Examples for Contract Testing

One of the most powerful features of Specmatic is external examples. While inline examples in the OpenAPI spec are useful for documentation, external examples allow us to define explicit test scenarios in standalone JSON files — giving us fine-grained control over what the contract tests validate.

Convention: The _examples Folder

Specmatic follows a naming convention: for each specification file foo-service.yaml, it automatically looks for a folder named foo-service_examples/ in the same directory. Each JSON file in this folder represents one test scenario with a full http-request / http-response pair. 

contracts/
├── user-service.yaml
├── user-service_examples/
│   ├── create_user_success.json        ← POST /users → 201
│   ├── get_user_by_id.json             ← GET /users/1 → 200
│   ├── list_users.json                 ← GET /users → 200
│   ├── get_user_not_found.json         ← GET /users/9999 → 404
│   └── create_user_missing_fields.json ← POST /users → 400
├── task-service.yaml
├── task-service_examples/
│   ├── create_task_success.json
│   ├── list_tasks.json
│   ├── list_tasks_by_status.json
│   └── create_task_missing_fields.json

External Example JSON Format

Each JSON file defines a complete request-response pair. When defining examples with dynamic formats, Specmatic provides specific built-in Matchers. Our CI validation highlighted this powerfully: using generic (string) wildcards fails validation for formatted strings. Instead, Specmatic requires its specific format Matchers like (email) or (datetime), maintaining structural flexibility without needing hardcoded values. Replacing generic string placeholders with these precise Specmatic Matchers successfully resolved our CI pipeline failures. 

// contracts/task-service_examples/create_task_success.json
{
  "http-request": {
    "method": "POST",
    "path": "/tasks",
    "headers": { "Content-Type": "application/json" },
    "body": {
      "title": "Implement login API",
      "description": "Build the authentication endpoint with JWT",
      "assigneeId": 1,
      "priority": "high"
    }
  },
  "http-response": {
    "status": 201,
    "headers": { "Content-Type": "application/json" },
    "body": {
      "id": "(number)",
      "title": "Implement login API",
      "description": "Build the authentication endpoint with JWT",
      "assigneeId": 1,
      "status": "pending",
      "priority": "high",
      "createdAt": "(string)"
    }
  }
}

Why external examples matter: Inline examples in the OpenAPI spec are great for API documentation, but they couple your test data to your specification. External examples let teams maintain test scenarios independently — including negative test cases (missing fields, invalid values) that are awkward to embed in the spec itself.

Validating Examples Against Specs

Specmatic provides a examples validate command that checks whether all external example files conform to the corresponding OpenAPI specification. If any example has the wrong field types, missing required fields, or invalid enum values, the validation fails immediately — catching spec drift before tests even run. 

# Validate all external examples against their specs
npx specmatic examples validate --spec-file contracts/user-service.yaml
npx specmatic examples validate --spec-file contracts/task-service.yaml
npx specmatic examples validate --spec-file contracts/notification-service.yaml
npx specmatic examples validate --spec-file contracts/analytics-service.yaml

CI/CD Integration with Specmatic

A contract validation pipeline is only valuable if it runs automatically on every push. We integrated Specmatic into our GitHub Actions CI/CD pipeline with a multi-stage workflow that validates specifications, runs contract tests against live services, and validates external examples — all without requiring any API keys or manual intervention.

Pipeline Architecture

Our CI pipeline is split into focused, independent jobs that run in sequence:

Stage What It Does Failure Means
1. Validate Specs & Examples Lints OpenAPI contracts with Redocly, validates external examples with Specmatic Spec has structural errors or examples are out of sync
2. Contract Tests Starts all 4 microservices, waits for health checks, runs specmatic test against each Service implementation doesn't match contract
3. Build Dashboard Compiles the React research dashboard Build or type errors in frontend code
4. Report Generates a summary in the GitHub Actions UI —

Key CI Design Decisions

  • Health-check polling instead of sleep: Rather than blindly sleeping for a fixed time after starting services, the pipeline polls each service's /health endpoint with retries. This is more reliable and eliminates flaky failures from slow service startup.
  • External examples validation as a gate: Before running expensive contract tests, the pipeline first validates that all external examples conform to their specs. This catches simple structural issues early and fast. Our recent CI runs highlighted the strictness of this step—it correctly failed our build when we incorrectly used generic (string) placeholders for formatted fields. Switching to precise Specmatic Matchers like (email) and (datetime) proved the framework's effectiveness in enforcing format-compliant, dynamic test payloads.
  • Resiliency via CLI flag, not config file: Resiliency tests are enabled for all services using the --resiliency CLI flag on each specmatic test invocation. We initially tried placing resiliencyTests: all inside the Specmatic v3 config's settings block, but this caused specmatic examples validate to crash because the MockSettings class does not recognize the resiliencyTests field. The CLI flag approach is cleaner: resiliency is only enabled during contract testing, never during example validation or stub generation.
  • V2-style CLI over v3 config for CI: Our CI uses explicit --spec-file and --testBaseURL arguments (v2-style) instead of Specmatic's v3 auto-discovery config. We discovered that the v3 config's automatic SUT/dependency management (starting mock servers, binding ports) conflicts with manually started services in CI. The v3 config is preserved as specmatic-config-example.yaml for reference.
  • No API key dependency: The CI pipeline only runs Specmatic contract tests — it does not run the LLM agent experiment (which requires an NVIDIA API key). This keeps CI green and reproducible without secrets.
  • Java requirement: Specmatic is a JVM-based tool. The pipeline uses actions/setup-java with Temurin JDK 17 to ensure the Specmatic CLI works correctly.
  • License via GitHub Secrets: The Specmatic Enterprise license is gitignored and not committed to the repo. Instead, the license content is stored as a GitHub Repository Secret (SPECMATIC_LICENSE) and written to license.txt at CI runtime. This enables resiliency tests in CI without exposing the license in version control.
  • Explicit yaml dependency: The @redocly/cli depends on redoc, which requires the yaml npm package at runtime. With npm workspaces, this transitive dependency may not be hoisted to the root node_modules, causing a MODULE_NOT_FOUND crash during redocly lint. Adding yaml as an explicit devDependency in the root package.json resolves this reliably.

Workflow Configuration

# .github/workflows/contract-validation.yml (excerpt)
jobs:
  validate-specs:
    name: Validate API Contracts & Examples
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - uses: actions/setup-java@v4
        with: { distribution: 'temurin', java-version: '17' }
      - run: npm install
      - name: Validate OpenAPI contracts (lint)
        run: |
          npx @redocly/cli lint contracts/user-service.yaml
          npx @redocly/cli lint contracts/task-service.yaml
          # ... remaining specs
      - name: Validate external examples
        run: |
          npx specmatic examples validate --spec-file contracts/user-service.yaml
          npx specmatic examples validate --spec-file contracts/task-service.yaml
          # ... remaining specs

  contract-tests:
    needs: validate-specs
    steps:
      - name: Write Specmatic license
        if: ${{ secrets.SPECMATIC_LICENSE != '' }}
        run: echo "${{ secrets.SPECMATIC_LICENSE }}" > license.txt
      # Start services → poll health → run specmatic test
      - name: Wait for services to be ready
        run: |
          for port in 3001 3002 3003 3004; do
            for i in $(seq 1 30); do
              curl -s http://localhost:$port/health && break
              sleep 1
            done
          done
      - run: npx specmatic test contracts/user-service.yaml \
               --testBaseURL=http://localhost:3001 --resiliency
      # ... remaining services

Lesson learned: Running Specmatic in CI caught issues we missed locally — particularly around Content-Type header enforcement and edge cases in query parameter validation. CI is the true litmus test for contract compliance.

Debugging CI Failures: Three Root Causes

Getting the CI pipeline to pass green was itself a valuable exercise in understanding Specmatic's internals. We encountered three distinct failure modes, each teaching us something about the tool's architecture:

  1. MODULE_NOT_FOUND: yaml — npm workspace hoisting issue
    The @redocly/cli depends on redoc, which transitively depends on yaml. With npm workspaces, yaml gets installed deep in node_modules/@redocly/cli/node_modules/redoc/node_modules/swagger2openapi/node_modules/yaml/ — but redoc.lib.js calls require('yaml') at the top level, expecting it to be hoisted. The fix was adding yaml as an explicit devDependency in the root package.json. This is a common npm workspace gotcha that doesn't appear in single-package projects.
  2. UnrecognizedPropertyException: resiliencyTests — v3 config MockSettings limitation
    We placed resiliencyTests: all under each service's settings block in specmatic.yaml. While this setting is valid for test-mode services, Specmatic's examples validate command loads the same config and treats all services as mocks. The MockSettings class has only 7 known properties (startTimeoutInMilliseconds, generative, delayInMilliseconds, etc.) — resiliencyTests is not one of them, causing a deserialization crash. The fix was removing resiliencyTests from the config and passing --resiliency as a CLI flag during specmatic test invocations only.
  3. Connection refused — v3 config auto-loading conflicts with v2 CLI
    When specmatic test --spec-file X.yaml --testBaseURL=http://localhost:3001 runs, it still auto-loads specmatic.yaml from the working directory. The v3 config declares a SUT (task-service) and dependency mocks (user-service mock on port 9001, notification-service mock on port 9003). Specmatic's internal orchestration attempted to manage these mocks, which interfered with the explicitly started user-service on port 3001 — causing connection refused errors on every test. The fix was renaming specmatic.yaml to specmatic-config-example.yaml to prevent auto-loading, since our CI uses v2-style explicit CLI arguments that don't need the v3 config.

After these three fixes, the full CI pipeline passes: 4 Redocly lint validations, 15 external example validations (across 4 services), 24 contract tests (9 user + 6 task + 6 notification + 3 analytics), and a successful dashboard build.

Specademy Learnings

Specademy Certificate of Completion — to be uploaded separately.

The Specademy certification program provided structured, hands-on training in contract-driven development. These are the key takeaways that directly influenced our project architecture:

  1. Contract-First Development: Writing the OpenAPI specification before any service implementation ensures all teams (and agents) share the same truth. This eliminates the "who changed the API?" problem entirely.
  2. Generative Testing is Not Optional: Specmatic's ability to auto-generate test cases from the spec — including negative tests, boundary conditions, and type mutations — is far more comprehensive than hand-written tests. The Specademy curriculum emphasizes this as the core value proposition.
  3. Stub/Mock Servers from Specs: Specmatic can spin up mock servers directly from OpenAPI specs, enabling frontend and backend teams to develop in parallel. In our multi-agent context, this means the Frontend Agent can test against a Specmatic stub while the Backend Agent implements the real service.
  4. Backward Compatibility Checks: Specmatic can compare two versions of a spec to detect breaking changes (removed endpoints, changed types, narrowed enums). This is critical for safe API evolution and should be integrated into every PR pipeline.
  5. External Examples as Living Test Suites: The _examples convention transforms test data management. Examples are version-controlled alongside specs, validated on every CI run, and can be shared across teams as the canonical test dataset.

Recommendation: For any team adopting Specmatic, the Specademy course is the fastest path to understanding contract-first workflows. The hands-on labs on resiliency testing and stub generation were particularly valuable for our multi-agent research.
